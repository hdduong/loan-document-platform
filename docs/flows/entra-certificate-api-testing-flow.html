Entra certificate client across Azure and AWS

An automated caller uses a certificate-signed client assertion to obtain an app-only PRODUCT TOKEN. If Postman is used, a certificate-capable helper or runner should inject only that short-lived token; the collection does not store a client secret, private key, token, PDF, or signed URL. A machine test proves the app-only path. It does not prove browser PKCE, Conditional Access, delegated scopes, UI CORS, or MSAL renewal.

One-time scripted setup

  • Product identity: register a per-environment Entra service client, upload only its CA-issued public certificate, assign the required application roles, grant consent, and add its client ID to the Azure ingress/API allowlists. The production example does not create this optional client by default. A production integration uses its own principal and certificate lifecycle, never a shared test identity.
  • Azure runtime identity: attach a dedicated user-assigned managed identity to the Azure API. It alone receives the federation application's AwsFederation.AssumeRole role; no runtime client secret or AWS access key exists.
  • AWS trust: configure an IAM OIDC provider for https://sts.windows.net/<tenant-guid>/. The runtime role pins aud = api://<federation-application-guid> and sub = <managed-identity-principal-object-id> exactly.
  • AWS permissions: restrict the role to the named DynamoDB table and indexes, exact quarantine/artifact/manifest S3 prefixes, KMS use through S3 for those objects, and the named upload-processor reconciliation function. It cannot call AppSync, the optional Jobs REST API, an IDP state machine, IAM, or deployment APIs.

Test runner to Microsoft Entra ID. 1 Client credentials: scope=api://<product-api-client-guid>/.default plus a private_key_jwt client assertion signed by the service client's private certificate.

Microsoft Entra ID to test runner. 2 PRODUCT TOKEN with aud=<product-api-client-guid>, assigned roles, idtyp=app, azpacr=2, service-principal oid, and no scp. Honor its actual expiry and refresh early.

Test runner to Azure API. 3 POST /v1/loans with Bearer token, a new create-loan Idempotency-Key, correlation ID, and {"loanId":"23051"}.

Azure verifies the app-only caller Signature, exact v2 tenant issuer, bare product API GUID audience, lifetime, tenant, immutable actor, allowed client, idtyp=app, azpacr=2, and the route's application role. Invalid authentication is 401; a valid token that fails policy is 403. If ingress rejects first, its body need not be the application's canonical problem shape.

Azure API to Microsoft Entra ID. 4 On the first AWS-backed call, or before cached credentials expire, the managed identity requests a dedicated v1 WORKLOAD TOKEN. The caller's product token stays at the Azure boundary.

Microsoft Entra ID to Azure API. 5 Workload token with issuer https://sts.windows.net/<tenant-guid>/, audience api://<federation-application-guid>, and subject equal to the managed identity's principal/object ID.

Azure API to AWS STS. 6 AssumeRoleWithWebIdentity presents the workload token; the call itself requires no pre-existing AWS credentials.

AWS verifies the Azure workload The IAM OIDC provider validates the v1 issuer/JWKS and the role's exact audience and subject conditions. A product token fails this trust. STS credentials are bounded by configured session and token lifetimes, cached only in API memory, and refreshed early.

AWS STS to Azure API. 7 TEMPORARY AWS CREDENTIALS for the one exact runtime role. They are never returned to the caller.

Azure API to private AWS. 8 Create the active loan registry item with a server-issued immutable loanInstanceId using a conditional DynamoDB transaction.

Azure API to test runner. 9 201 with loanId=23051 and the generated loanInstanceId.

Test runner to Azure API. 10 POST /v1/loans/23051/documents with a new create-document idempotency key and fileName, PDF content type, byte size, and Base64 SHA-256.

Azure API to private AWS. 11 Transactionally register the stable documentId and physical uploadId; create an exact-key, PDF-only, size/checksum/KMS- constrained presigned POST. Default lifetime is 600 seconds and is capped by the remaining STS credential lifetime.

Azure API to test runner. 12 201 with documentId, uploadId, AWAITING_UPLOAD, expiry, and PRESIGNED POST. Never AWS credentials.

Test runner directly to Amazon S3. 13 Multipart POST every returned form field and append the PDF last. Deliberately send no Entra bearer token; S3 enforces the signed policy and configured CORS.

Test runner to Azure API. 14 POST .../uploads/{uploadId}/complete with its own completion-intent key and an empty body. Response 202 pins one exact S3 VersionId and reports VALIDATING. A retry reuses this completion key; it does not upload the PDF again.

Asynchronous convergence inside AWS: object creation starts GuardDuty malware scanning independently of completion. The processor proceeds only when the same pinned version is client-complete and NO_THREATS_FOUND, then verifies checksum, size, encryption, metadata, PDF structure, and page limit. Persisted normal states are AWAITING_UPLOAD → VALIDATING → QUEUED → SCREENING → EXTRACTING → SUCCEEDED. Selection is committed atomically between screening and extraction; terminal alternatives are HOLD, REJECTED, and FAILED.

Test runner to Azure API. 15 Poll GET .../documents/{documentId}. On success, request bounded inline data points or a fresh source/selected/data-points download grant. Every Azure call is re-authorized for its route.

Azure API to test runner. 16 Inline JSON or DOWNLOAD GRANT with expiresAt. The default signed-URL lifetime is 120 seconds, also capped by the remaining STS lifetime.

Test runner directly to Amazon S3. 17 GET the exact-version artifact directly from S3 without an Entra token. A signed URL can be reused until expiry; request a fresh grant after expiry.

Archive and replacement continuation

1 · Archive a terminal document

POST /v1/loans/{loanId}/documents/{documentId}/archive with a new document-archive intent key. The immutable snapshot receives display alias documentId_001; the logical documentId remains stable.

2 · Upload a replacement

POST .../documents/{documentId}/uploads creates a new uploadId and repeats upload, completion, and processing. Its later archive becomes _002.

3 · Archive the loan

When no owned document is incomplete or processing, POST /v1/loans/{loanId}/archive creates an immutable manifest for every document and exact artifact version. It does not move, rename, or copy S3 objects. The first loan alias is 23051_001.

4 · Recreate the business key

A later POST /v1/loans with 23051 creates a new loanInstanceId. Archiving it allocates 23051_002; prior archives remain read-only.
