Entra interactive sign-in across Azure and AWS

The SPA uses single-tenant authorization code with PKCE and has no secret or certificate. A delegated USER TOKEN must contain both the route's delegated scp value and its matching assigned roles value. UI capability checks improve usability; the Azure API remains the authoritative control plane. The browser never calls AppSync, an AWS Loan API, DynamoDB, or an IDP workflow.

Browser and registration baseline

  • Load and schema-validate /runtime-config.json before rendering. Tenant, client, API audience, API base URL, scopes, redirect URI, and logout URI must match the deployed environment; invalid configuration fails closed.
  • Register exact HTTPS redirect/logout URIs and no wildcard. Production has no localhost redirect. Configure MSAL's cache location as sessionStorage; application code does not manually persist tokens.
  • Assign users/groups the route roles they need. In production, the API requires a delegated scope and the matching role. The SPA client ID is explicitly allowlisted at ingress and in the application validator.
  • Restrict S3 CORS to the exact deployed SPA origin and only the methods/headers required by the returned upload/download grants.

0 · App boot Validate runtime configuration before constructing MSAL or an API client. Preserve the requested route for a post-login return.

Browser to Microsoft Entra ID. 1 Authorize redirect: MSAL creates a PKCE verifier/challenge and state, then requests the required delegated scopes. PKCE binds the authorization code to the SPA instance holding the verifier.

Entra authenticates and authorizes the user The configured authentication method, MFA, Conditional Access, assignment, and consent run on Microsoft's origin. The SPA never sees a password or other primary credential. Failure returns no authorization code or product token.

Microsoft Entra ID to browser. 2 Entra redirects to the exact registered callback with a short-lived authorization code and state; MSAL validates state.

Browser to Microsoft Entra ID. 3 MSAL exchanges the authorization code plus PKCE verifier. No SPA secret or certificate is involved.

Microsoft Entra ID to browser. 4 Short-lived USER TOKEN with aud=<product-api-client-guid>, delegated scp, assigned roles, user oid, and azp=<spa-client-guid>. MSAL owns its tab-scoped sessionStorage cache.

5 · UI capability gate Show an action only when the token exposes both its scope and role. This is cosmetic; never treat hidden controls as authorization.

Browser to Azure API. 6 POST /v1/loans with Bearer token, a new create-loan intent key, a correlation ID, and {"loanId":"23051"}.

Azure verifies the delegated caller Signature, exact v2 issuer and bare API GUID audience, lifetime, tenant, immutable user actor, allowlisted SPA client, route scope P, and matching assigned role P.Role. Invalid authentication is 401; an authenticated user missing either permission claim is 403.

Azure API to Microsoft Entra ID. 7 On demand, the Azure managed identity requests its separate v1 WORKLOAD TOKEN for the dedicated federation application. The user's token is never forwarded to AWS.

Azure API to AWS. 8 Exchange the workload token through AssumeRoleWithWebIdentity, cache the returned TEMPORARY AWS CREDENTIALS in API memory, and use the narrowly scoped role for the DynamoDB transaction.

AWS verifies the Azure workload, not the user The OIDC provider and role trust pin the v1 issuer, federation application audience, and managed-identity principal/object subject. The product token has the wrong issuer, audience, and subject for this trust boundary.

Azure API to browser. 9 201 with the caller-owned loanId and server-issued immutable loanInstanceId.

Browser to Azure API. 10 Compute Base64 SHA-256, then POST /v1/loans/23051/documents with file name, PDF content type, byte size, checksum, and a new create-document intent key.

Azure API to browser. 11 201 with stable documentId, physical uploadId, AWAITING_UPLOAD, expiry, and an exact constrained PRESIGNED POST. No AWS credentials are exposed.

Browser directly to Amazon S3. 12 Multipart POST every returned field, append the PDF last, and show progress. Send no Entra bearer token; the signed form and exact-origin CORS govern this one S3 operation.

Browser to Azure API. 13 POST .../uploads/{uploadId}/complete with a new completion-intent key and empty body. If completion times out, reuse this completion key and offer Retry completion—never upload the PDF again. The create-document key is different.

Asynchronous processing: S3 object creation and completion may arrive in either order. Only the exact pinned, client-complete, malware-clean version advances through validation, text-only screening, deterministic CD selection, selected-page materialization, and full extraction. Persisted normal states are AWAITING_UPLOAD → VALIDATING → QUEUED → SCREENING → EXTRACTING → SUCCEEDED; terminal alternatives are HOLD, REJECTED, and FAILED.

Browser to Azure API. 14 Poll GET .../documents/{documentId} near two seconds, back off to 15 seconds, pause while hidden, resume on focus, abort on navigation, and stop on a terminal state. Call acquireTokenSilent before API requests as needed.

Browser to Azure API. 15 Render bounded inline data points as escaped read-only JSON, or request a fresh source/selected/data-points download grant. The API rechecks Document.Read or DataPoints.Read.

Azure API to browser. 16 DOWNLOAD GRANT for one exact S3 object version, with expiresAt and a no-store download response policy.

Browser directly to Amazon S3. 17 Open or save the requested artifact directly from S3 without an Entra token. The signed URL remains reusable until expiry; use noopener,noreferrer for a new browsing context, and request a fresh grant for a later download.

Archive and replacement experience

Archive a document version

Confirm read-only semantics, then submit a distinct document-archive intent. The first immutable snapshot is displayed as documentId_001. A replacement retains the stable documentId, receives a new uploadId, and can later become _002.

Archive the active loan

Explain that all owned documents are included. A processing or incomplete document blocks the operation with 409. Success creates a logical immutable manifest; it does not move S3 objects. The first display alias is 23051_001.

Recreate the base loan ID

Creating 23051 after archive returns a new loanInstanceId. Its eventual archive is 23051_002; current and archived instances remain separately addressable.

UI security and resilience details

  • Token renewal: call acquireTokenSilent before protected requests. On 401, force a fresh acquisition and retry once. If MSAL reports interaction_required, redirect and return to the requested route. SPA code never directly handles a refresh token.
  • Idempotency: create one key per user intent, disable double submission, and reuse only that operation's key across timeout/offline/5xx retry. A genuinely new action receives a new key.
  • Storage: MSAL alone owns its tab-scoped sessionStorage token cache. App code does not persist tokens, grants, PDFs, or data points in localStorage, IndexedDB, service workers, logs, analytics, or error telemetry. A user-requested browser download is intentional.
  • Audit: createdBy, archivedBy, and idempotency ownership use the validated immutable Entra oid. A correlation ID joins sanitized diagnostics; it does not identify a person by itself.
  • Accessibility: status is never color-only; all actions are keyboard reachable; focus, labels, live regions, validation, and confirmation copy meet WCAG 2.2 AA.
